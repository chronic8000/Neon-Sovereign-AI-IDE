## 🏢 The Software House Model (Level 5 Autonomy)

Neon Sovereign doesn't wait for your approval. It gives you a **team** operating through a high-speed **Ephemeral Gossip Protocol** (in-memory SQLite), allowing agents to dynamically negotiate interface contracts before generating muscle code. Work flows through a defined, autonomous structure:

1. **The Brief:** You provide a detailed prompt, hit execute, and step away.
2. **Architecture:** The Architect designs the system and breaks it into parallel workstreams via AST-Pointer UUIDs, drastically shrinking context windows.
3. **UX Blueprint:** The UX Specialist generates the Spatial Constraint Graph — the mathematical blueprint for any UI layout involved.
4. **Execution:** Specialist Developers execute workstreams in parallel. They communicate via the Ephemeral Gossip Protocol (SQLite), leaving notes for each other on the Blackboard to prevent interface clashes.
5. **The Classroom (Peer Review):** Developers swap AST deltas and cross-check interfaces *before* handing it to QA.
6. **The Verification Gauntlet:** The Forge Master compiles the code in the Shadow Buffer. The Geometry Auditor and Silicon Retina verify the UI. The Chaos Agent attempts to break it. The Autopsy Agent rewinds any crash.
7. **Sign-Off & Launch:** The Sentinel (Tech Lead) grades the final output, commits it to the codebase, and launches the application in the SpiceViewport.

```
You give a task
       ↓
Architect designs the system → produces a JSON Task DAG with AST UUID pointers
       ↓
UX Specialist generates the Spatial Constraint Graph (if UI is involved)
       ↓
Kahn's Algorithm calculates optimal parallel execution order
       ↓
Specialist Developers execute workstreams in parallel (negotiating via Ephemeral DB)
       ↓
Peer-Review Phase: Agents cross-check interfaces and resolve clashes
       ↓
Forge Master compiles in the Shadow Buffer (injecting DPAPI credentials safely)
       ↓
The Verification Gauntlet (Silicon, Protocol, Stability, and Multimodal UI Audits)
       ↓
Sentinel signs off → commits codebase → launches application in SpiceViewport
       ↓
You return to a running app, a proof-of-work screenshot, and a complete audit log.
```

2. **Forge Master** provisions the RISC-V GCC cross-compile chain via `ToolRegistry` automatically.
3. **Developers** write the ISR and DMA routines in parallel.
4. **Binary Analyst** runs Ghidra Headless against the ELF, verifying interrupt vector alignment at the silicon level.
5. **SpiceViewport** boots a QEMU RISC-V guest to execute and observe the firmware in real-time.
6. **Sentinel** audits `volatile`-correctness on all memory-mapped registers.
7. The Swarm commits the code and leaves the emulator running for the Handoff.

**🎯 Task 3:** *Full-stack web app — TypeScript React frontend, Go REST API, PostgreSQL.*
1. **Architect** defines the service boundary DAG.
2. **UX Specialist** generates the Spatial Constraint Graph for the React UI layout.

4. **Geometry Auditor** extracts the OS accessibility tree and proves the UI layout mathematically matches the Spatial Constraint Graph.
5. **Silicon Retina (VLM)** captures a raw framebuffer frame, flags a low-contrast button overlap, and leaves a red-boxed screenshot on the Blackboard for the Developers to fix.
6. **Network Scout** verifies CORS headers via TShark packet inspection on the Shadow Buffer's loopback interface.
7. **Sentinel** approves the final build. The Swarm launches the web app, captures a final screenshot, and presents the live application at the Handoff.

## 🧠 Hardware Requirements & VLM Economics

This system is **not lightweight by design.** Running the Gemma-4 26B Architect, the Gemma-4 E2B Sentinel, a local Vision-Language Model, a 120FPS Vulkan renderer, and a live SpiceViewport simultaneously on a single GPU requires serious hardware — and a hard budget enforcer, not runtime negotiation.

| Component | Minimum (Degraded / Sentinel-Only) | Recommended (Full Swarm) | Notes |
| :--- | :--- | :--- | :--- |
| **CPU** | 8-core (Ryzen 3700X / i7-9700K) | 16-core+ (Ryzen 7950X / i9-13900K) | Agent parallelism, shadow compiles, MCTS rollouts |
| **RAM** | 32GB | 64GB+ | Prevents context thrashing across parallel workstreams |
| **GPU** | 12GB VRAM | 24GB VRAM (RTX 3090/4090+) | LLM inference, Vulkan renderer, SpiceViewport, VLM |
| **Storage** | SSD | NVMe Gen4 | ProjectLedger indexing, shadow buffer, build artefacts |
| **OS** | Windows 11 + WSL2 | Same | Native IOCP, DPAPI, UIAutomation, WSLManager |

### ⚠️ VRAM Reality Check, MoE Economics & Static Partitioning

VRAM is **statically partitioned at startup** from `config/neural.toml`. The four consumers — inference, renderer, SpiceViewport, and system reserve — never bid against each other at runtime. The Governor enforces hard budgets before inference begins, meaning the system never discovers it has run out of VRAM mid-inference.

> **Critical architecture note:** The Gemma-4-26B-A4B model is a **Mixture-of-Experts (MoE)** architecture. Despite the 26B total parameter count, only **~4B parameters are active per forward pass**. This has two important implications that distinguish it from a dense 26B model:
>
> - **Full weight load at Q4_K_M is ~14–15GB VRAM** — approximately 1–2GB less than a comparable dense 26B model, meaningfully improving headroom on both tiers.
> - **Per-token inference compute cost is equivalent to a ~4B dense model**, delivering 26B-class reasoning at a fraction of the generation latency and power cost.
>
> This MoE characteristic is why the Full Swarm tier is achievable on 24GB rather than requiring 40GB+, and why the 12GB Degraded tier is more viable than the raw parameter count implies.

**Static VRAM partition (24GB Full Swarm example):**

| Consumer | Priority | Budget | Behaviour at Limit |
| :--- | :--- | :--- | :--- |
| **Inference** (Architect + Sentinel) | 1 — Highest | 18GB | Re-quantize or CPU-offload the batch |
| **Renderer** (Vulkan/DirectX) | 2 | 2GB | Reduce framebuffer resolution |
| **SpiceViewport** (QEMU guest) | 3 | 2GB | Drop SPICE stream resolution or cap FPS |
| **System reserve** | — | 2GB | Driver overhead, OS, headroom |

> The SpiceViewport is deliberately the lowest priority consumer. It is an observability surface — dropping its frame rate or resolution during a heavy swarm task has zero operational impact on the guest execution or the agents.

**Compilation Mode** is the worst-case scenario: the Forge Master is hammering CPU/RAM with parallel compile jobs while the Swarm remains active. When `build_project` is called, the Governor automatically engages `[compilation_mode]`:
- Inference batch size is halved — the Swarm stays alive but consumes less VRAM pressure per cycle
- Non-critical background agents are suspended until the build completes
- The SpiceViewport is frozen to a static thumbnail — zero GPU cost
- Full budgets are restored the moment `build_project` returns

#### 12GB Tier (Degraded Mode)

You cannot run the full swarm locally on 12GB. The system automatically degrades in one of three ways:
- Runs the 2B Sentinel only, with the Architect offloaded to CPU (incurring significant latency)
- Runs a heavily quantized Architect with the Sentinel as the primary quality gate
- Routes Architect inference to a hybrid API endpoint while keeping verification local

Thanks to the MoE architecture, the Architect's ~14–15GB full weight load means a future 16GB tier (RTX 4080 / RX 7900 XTX) will run the full swarm with modest headroom — this tier is not as far out of reach as a dense 26B model would imply.

*VLM Recommendation:* Use **LLaVA 1.5 (7B, Q4_K_M)** for the Silicon Retina. At ~5GB VRAM, the system runs it serially — pausing the Architect and swapping the VLM into VRAM only when the Geometry Auditor triggers the visual gate.

#### 24GB Tier (Full Swarm Mode — Recommended)

Runs the Gemma-4-26B Architect, Gemma-4-E2B Sentinel, Vulkan UI, VLM, and SpiceViewport concurrently with zero compromises. No swapping. No throttling. No serial serialisation of inference.

*VLM Recommendation:* Use **Pixtral 12B** or **LLaVA v1.6 34B (Quantized)**. These provide superior aesthetic reasoning and spatial awareness for the Silicon Retina pipeline, co-residing seamlessly with the rest of the neural stack within the static partition.

## ⚡ Core Architecture & Persistent Memory

The Swarm never forgets, and it never hallucinates context. Intelligence does not begin with the LLM — fast, deterministic algorithms handle reasoning *before* any model is invoked.

### 🧩 The Non-LLM Brain — Deterministic CPU Offloading

- **Kahn's Algorithm (`BlackboardManager`)** — Topological sorting of the Task DAG. The swarm never guesses execution order; it calculates the mathematically optimal parallel path.
- **Ephemeral Gossip Protocol (`m_ephemeral_db`)** — Agents do not pass massive code blocks to each other. They negotiate interface contracts dynamically via an in-memory SQLite database before writing muscle code, drastically shrinking context windows and preventing hallucinations at the source.
- **Omniscient Terminal Ledger** — The `TerminalManager` streams all stdout/stderr directly into the SQLite `ProjectLedger`. If a build fails at 3am, the agents don't guess why when they retry; they query the ledger to see the exact compiler error, the exact line, and the exact prior fix attempt.
- **The Blackboard** — A shared memory space where agents leave typed artefacts for each other: the Geometry Auditor deposits spatial coordinates, the Sentinel deposits audit grades, the Autopsy Agent deposits stack traces, and the Chaos Agent deposits crash reports — all persisted and queryable.
- **Force-Directed MCTS Visualization** — The `Analyst` agent executes genuine Monte Carlo Tree Search (MCTS) — running simulated rollouts, UCB1 node selection, and backpropagation to evaluate architectural branches before committing to a plan. The reward signal is a composite of: historical BM25 similarity from past successful fixes, projected token-context footprint, and statically analyzed compiler pass probability *(derived dynamically from Tree-sitter cyclomatic complexity and historical linker-fault heuristics)*. This reasoning tree is dynamically mapped in the MDI UI using a real-time Spring-Embedding algorithm — you can watch the Swarm think.
- **BM25/TF-IDF Memory (`ProjectLedger`)** — Historical fixes, toolchain quirks, and interface contracts stored in SQLite and retrieved via a native C++ BM25 ranker. O(1) lookup. No vector embeddings. No GPU cost. The longer you use it, the faster the Swarm resolves novel problems.
- **Tree-sitter Complexity Router (`ASTContextManager`)** — Cyclomatic complexity > 15 automatically escalates to the 26B Architect. Simpler tasks route to the 2B Sentinel model. Zero wasted VRAM on trivial work. Operates across any Tree-sitter-supported language grammar.
- **Chase-Lev Work-Stealing Deques** — Every agent manages its own lock-free deque. If an agent becomes idle, it autonomously steals tasks from the back of other agents' queues, ensuring zero idle cycles during massive builds or refactors.
- **VCG (Vickrey-Clarke-Groves) Auction Loop** — Access to execution lanes is governed by a micro-auction. Agents bid based on task complexity and urgency; the VCG mechanism ensures the most critical engineering tasks are allocated the highest-performing hardware lanes without resource contention.
- **NUMA-Aware Core Pinning** — The Architect is locked to high-performance P-cores with dedicated cache lanes. Sentinel auditors are distributed across E-cores to preserve UI responsiveness. Role-based hardware affinity formalised in the SDK.

### 🐝 The Infinite Swarm — Role-Locked Specialisation

Every agent operates under hard, non-negotiable boundaries enforced by the C++ core:

| Agent | Role | Hard Constraint |
| :--- | :--- | :--- |
| **The Architect** | Designs the system via AST UUID pointers | Barred from writing implementation code |
| **The Developers** | Execute parallel implementation workstreams | Cannot modify the overall DAG or task structure |
| **The Forge Master** | DevOps scope (Builds, NDK, ADB pipelines) | Barred from mutating application logic; strictly manages build scripts, linkers, Dockerfiles, and deployment |
| **The Binary Analyst** | Ghidra Headless + EGL/VirtIO Audits | Strictly read-only analysis; cannot alter source code |
| **The UX Specialist** | Defines spatial constraint blueprints | Generates mathematical bounds only; barred from writing UI framework code |
| **The Autopsy Agent** | Rewinds execution state via `rr` / eBPF | Forensic extraction only; read-only access to crash dumps and registers |
| **The Chaos Agent** | Agentic fuzzing of SPICE queues & UI | Destructive testing only; routes crashes to the Autopsy Agent to mandate Developer fixes |
| **The Sentinel** | Adversarial code review & Multimodal QA | Final sign-off authority; auditing and grading only; never generates or patches code |

## 👁️ The Silicon Retina — Absolute Visual & Spatial Verification

Current AI IDEs are blind text-calculators. Neon Sovereign **looks at what it builds and tests it with its own hands.**

1. **Synthetic ID Injection:** Developer agents automatically stamp AST UUIDs into the compiled UI framework's accessibility tags.
2. **Geometry Auditor:** Using native Windows `UIAutomation` and Linux `AT-SPI`, the system extracts absolute `[X, Y, W, H]` coordinates from the rendered application, waiting for UI idle-state to prevent desync between layout and measurement.
3. **The Math Gate:** The Sentinel verifies the parsed coordinates against the UX Specialist's initial Spatial Constraint Graph. If the numbers don't match the blueprint, the build does not proceed.
4. **The VLM Retina:** If the math passes, a raw framebuffer screenshot is captured via GDI/WIC. A local Multimodal Vision-Language Model (Pixtral 12B on the 24GB tier; LLaVA 1.5 7B on the 12GB tier) audits the app for contrast violations, element overlaps, and aesthetic failures.
5. **Active HID Testing:** The Swarm doesn't just look. It autonomously sends synthetic mouse clicks and keystrokes directly into the SpiceViewport to ensure buttons respond, inputs don't crash the app, and interactive elements behave correctly under load.
6. **Red-Box Feedback:** Any spatial or visual failure results in a red-boxed screenshot deposited on the Blackboard and routed immediately back to the Developers for an autonomous refactor loop.

## 🗡️ The Verification Gauntlet: Hostile Engineering & Omniscience

Neon Sovereign doesn't just write polite code; it attacks it to guarantee stability before the Sentinel signs off.

- **Agentic Fuzzing (The Chaos Protocol):** After a successful build, the Chaos Agent takes over the QEMU/ADB bridge, rapidly resizing framebuffers, thrashing VirtIO queues, and spamming malformed HID packets to ensure the hypervisor doesn't break under adversarial conditions.
- **Time-Travel Autopsy:** If a kernel panic or JNI crash occurs across an exotic OS boundary, the Autopsy Agent uses `rr` (Record and Replay) and eBPF probes to rewind the execution trace. It injects the exact nanosecond of memory corruption back into the Swarm's context via the Blackboard, routing the failure directly to the Developers for a targeted fix — not a guess.
- **Micro-Architecture Optimizer:** High-complexity loops trigger the Swarm to write standard C++, ARM NEON, and aggressive `#pragma` variants in parallel. The Forge Master compiles them, benchmarks CPU cycles and cache misses via PDH telemetry, and silently commits the mathematically fastest variant. You get optimal silicon performance without asking for it.
- **Re-Fuzz After Fix:** After the Autopsy Agent mandates a refactor, the Forge Master recompiles and the Chaos Agent re-runs the full fuzzing suite against the patched target. The build does not proceed to the Sentinel until the exact failure mode that was reported has been provably eliminated.

- **Edge Forge (ADB Pipeline):** Autonomously compiles C++/Rust NDK backends, packages APKs, deploys via the Android Debug Bridge, and intercepts Logcat telemetry natively to trap JNI panics in real-time.

## 🛡️ Defensive Engineering & Safety

### 🔒 The Sovereignty Gate & The Rejection Path

When the Sentinel grades and approves code, it is committed to the codebase and the application is launched. In autonomous mode, this is the default flow.

**Tactical Override (The Rejection Path):** If you are watching the MDI dashboards and see the Swarm heading in the wrong direction, you can intervene at any time. Issue a voice command or open the override prompt: type a prose note explaining *why* the current direction is wrong (*"Memory bounds are safe, but I want standard library arrays instead of raw pointers"*). This operator mandate is injected directly into the Ephemeral DB as a High-Priority Correction, and the Swarm routes back to the Architect to re-plan the branch. ### 🔒 Crucible Shadow Buffer & AST Bounding
